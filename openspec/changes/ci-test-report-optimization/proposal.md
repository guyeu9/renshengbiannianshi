# CI测试报告优化提案

## Context
当前的CI测试报告使用dorny/test-reporter生成，虽然功能完整，但对技术小白用户不够友好。报告主要面向开发者，使用专业术语，缺乏直观的视觉提示和清晰的总结。

需要优化的问题：
1. 报告标识不够直观，难以快速识别异常
2. 缺乏总结性结论，用户需要自己分析才能知道整体结果
3. 测试模块覆盖范围不明确
4. "有问题"和"无问题"的测试项区分不够清晰
5. 使用过多专业术语，缺乏解释
6. 缺乏直观的视觉编码（颜色、图标等）
7. 包含过多技术细节和冗长日志

## Scope
### 涉及文件：
1. `.github/workflows/android-apk.yml` - CI工作流配置
2. 新增脚本：`life_chronicle/scripts/generate_friendly_test_report.dart` - 友好测试报告生成脚本
3. 新增目录：`openspec/changes/ci-test-report-optimization/` - 提案文档目录

### 改动范围：
- 在现有CI工作流中添加友好测试报告生成步骤
- 新增Dart脚本来处理原始测试结果并生成友好报告
- 报告将以HTML格式生成并作为工件上传

## Behavior

### 输入：
- Flutter测试产生的原始JSON报告 (`test-results.json`)
- 测试覆盖率数据 (`coverage/lcov.info`)

### 输出：
- 友好化的HTML测试报告，包含以下内容：

#### 1. 报告标识
- 如果有失败的测试，报告名称标注为"有问题 - Flutter测试报告"
- 如果全部通过，标注为"正常 - Flutter测试报告"

#### 2. 总结性结论（位于报告开头醒目位置）
- **整体状态**：用大图标和颜色展示（✅ 全部通过 / ❌ 存在问题）
- **通过率**：显示通过/总测试数的百分比
- **问题项数量**：明确指出有多少个测试失败
- **测试覆盖模块**：列出所有测试的功能模块

#### 3. 测试模块覆盖清单
- 清晰列出所有测试覆盖的功能模块：
  - 数据访问层（DAO）测试
  - 业务逻辑服务测试
  - 状态管理Provider测试
  - 工具类测试
  - 集成测试
  - 备份服务测试

#### 4. 测试结果分类展示
- **无问题的测试项**（绿色背景，✅图标）
- **有问题的测试项**（红色背景，❌图标，突出显示）

#### 5. 问题描述优化
- 使用通俗易懂的语言重述技术错误
- 必要时提供术语解释（如："断言失败" = "预期结果和实际结果不一致"）
- 只保留关键错误信息，删减冗长日志

#### 6. 视觉呈现
- 红色：表示有问题/失败
- 绿色：表示正常/通过
- 使用直观的emoji图标（✅❌⚠️📊）
- 关键信息使用大号字体加粗显示

## Risks
### 可能破坏什么：
1. **CI工作流稳定性**：新增步骤可能影响现有CI流程
   - 缓解措施：使用`if: always()`确保即使报告生成失败也不影响主流程
2. **报告生成时间**：新增脚本可能增加CI运行时间
   - 缓解措施：脚本设计简洁高效，预期增加时间<30秒
3. **依赖管理**：新增Dart脚本可能需要额外依赖
   - 缓解措施：仅使用Flutter SDK自带的核心库

### 风险等级：低
- 改动仅在CI流程中添加新步骤，不修改现有功能代码
- 报告生成作为可选步骤，不影响主构建流程
