分步“纯 AI 开发”实现方案（可执行清单）
下面按“你不会写代码、让 AI 一步步生成可运行工程”的方式拆解。每一步都以“可运行、可验证”为目标。

Phase 0：项目脚手架（1~2 天）
目标：Flutter 能跑、数据库能建、能插入/查询一条记录。

让 AI 生成：
Flutter 项目结构（feature-first）
Riverpod 基础模板
Drift + SQLite 初始化、迁移框架
你要验证：
安卓真机运行
插入一条 food 记录并查询显示
Phase 1：数据层与“万物关联”最小闭环（3~7 天）
目标：实现 6 大实体中的 2~3 个（如 food/joy/friend）+ link 表。

让 AI 生成：
表结构 + DAO + migrations
link 的增删查（按 entity 查关联）
验证：
发布美食时能关联朋友；详情页能跳转关联记录（PRD 关联展示）
人生编年史APP产品需求文档
Phase 2：记录发布与列表（瀑布流）+ 本地图片存储（5~10 天）
目标：可用的“记录-浏览-详情”体验。

让 AI 生成：
图片选择/压缩/缩略图
瀑布流列表、筛选（日期/标签/评分/地点等，PRD有多维筛选）
人生编年史APP产品需求文档
Phase 3：WebDAV 备份（全量 zip + 增量） （5~12 天）
目标：实现 PRD 的“wev网盘备份zip + 增量备份（json、图片）”
人生编年史APP产品需求文档

让 AI 生成：
change_log 记录机制（每次改动写日志）
打包：导出 JSON + 拷贝新增媒体 → zip → AES 加密 → WebDAV 上传
恢复：下载 → 解密 → 回放 change_log / 导入数据
验证：
两台安卓机能通过同一 WebDAV 做同步（至少单向恢复成功）
Phase 4：搜索（FTS5）+ 图片 OCR 入库（7~14 天）
目标：先把“可用的搜索”做出来（不做向量也能很强）。

让 AI 生成：
search_doc 生成规则（标题+标签+地点+朋友名+ocr_text）
FTS5 虚拟表 + 同步触发更新（或应用层维护）
OCR pipeline（拍照/选图后后台跑 OCR，写入 media.ocr_text）
验证：
输入“日料/张三/成都/2024年3月开心”等能快速命中 
人生编年史APP产品需求文档
Phase 5：年度报告（SQL 统计 + AI 文案） （10~20 天）
目标：做出 PRD 的年度报告 MVP 
人生编年史APP产品需求文档

让 AI 生成：
年度 stats 的 SQL 集合（每个模块一套查询）
year_item 生成器（限长、可追溯）

## ai管理的遗漏关注点提醒

### 1. API Key 加密存储

建议使用 `flutter_secure_storage` 进行 API Key 的加密存储，避免明文存储在数据库中。

### 3. 流式响应支持

AI 史官对话功能需要流式响应以实现打字机效果，不同 API 类型的流式实现方式不同。

### 5. 多模态支持

部分模型支持图片输入（如 Gemini Vision、Claude Vision），后续可扩展支持。

### 7. 与现有功能的集成顺序

建议按以下顺序集成：

1. 先完成 AI 模型管理功能
2. 再接入 AI 史官对话功能
3. 最后接入年度报告生成和智能搜索功能
- AI 史官对话页可读取当前激活的对话服务商配置
- 年度报告生成可读取当前激活的对话服务商配置
- 智能搜索意图解析可读取当前激活的对话服务商配置
- Embedding 向量生成可读取当前激活的 Embedding 服务商配置
