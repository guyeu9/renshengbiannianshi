核心原则：**数据全部在本地**，没有自建后端，没有云数据库。联网只做两件事：调 AI 大模型 API、调 Embedding API。同步只走 WebDAV 网盘（如坚果云）[1]。

## 二、技术选型清单

| 层级 | 选型 | 理由 |
|---|---|---|
| 跨端框架 | Flutter 3.x (Dart) | 一套代码先出 Android 后出 iOS；自绘引擎对 PRD 要求的大量动效（彩带、气泡、涟漪、骨架屏）实现成本最低 [1] |
| 状态管理 | Riverpod | 模块多、实体关联复杂，Riverpod 的 Provider 依赖树比 Bloc 更适合"万物互联"的数据流 |
| 本地数据库 | Drift (SQLite) + FTS5 | 关系型结构完美匹配 PRD 的六大实体关联模型 [1]；FTS5 提供全文检索能力 |
| 本地加密 | SQLCipher + Keystore/Keychain | PRD 要求"加密备份" [1]，数据库层面加密最彻底 |
| 地图 SDK | 高德地图 Flutter 插件（Android SDK 10.1.200） | 国内场景，味觉地图 + 旅行足迹地图 [1] |
| PDF 导出 | dart_pdf + printing | 人生传记 PDF、年度味觉地图长图、行程导出 [1] |
| EPUB 导出 | epubx (Dart) | 人生传记 EPUB 格式 [1] |
| 图片 OCR | Google ML Kit (离线) | 本地识别菜单/票据/招牌文字，不上传图片 |
| 图片标签 | Google ML Kit Image Labeling | 本地输出"海滩/寿司/咖啡"等标签 |
| EXIF 提取 | exif (Dart package) | 提取拍摄时间、经纬度、设备信息 |
| WebDAV 同步 | webdav_client (Dart) | 增量备份到坚果云等网盘 [1] |
| AI 大模型 | DeepSeek / 通义千问 API | 联网调用，用于意图解析、情绪总结、社交洞察、年度报告文案生成 |
| Embedding | bge-m3 API 或 text-embedding-3 | 联网调用，为记录生成向量，存本地做语义检索 |
| 本地向量存储 | SQLite BLOB 字段 | 向量存本地，查询时 Dart 侧做余弦相似度计算 |

### 2.x Windows 本地开发环境（Flutter）

- Flutter SDK：放置在 `_toolchain/flutter`，并设置国内镜像环境变量：
  - `PUB_HOSTED_URL=https://pub.flutter-io.cn`
  - `FLUTTER_STORAGE_BASE_URL=https://storage.flutter-io.cn`
- 关键前置条件（Windows 桌面）：需要启用系统“开发者模式”或使用管理员权限运行终端，否则无法创建符号链接，包含插件的 `flutter pub get / flutter run` 会失败（Flutter 会提示 “Building with plugins requires symlink support.”）。
- 编译级验证基线：当前仓库以 `flutter analyze` + `flutter test` 作为最小可重复验证；网络图片会在测试中使用 HttpClient 桩，避免测试运行时的网络依赖。
- 构建验证基线：当前环境已通过 `flutter build web`，可用于持续集成的首个稳定目标；Windows 桌面构建仍依赖本机安装 Visual Studio（Desktop development with C++）。
- CI 构建基线：仓库新增 GitHub Actions 流水线用于输出 Android APK，见 `.github/workflows/android-apk.yml`。

### 问题根因与修复（2026-02-23 CI 环境兼容性专项）

#### 1. CI 环境构建失败问题
- **现象**：CI 流水线（如 GitHub Actions）在预检或构建阶段失败，提示 `flutter.sdk` 未找到，或预检脚本报错。
- **根因**：
  1.  **local.properties 缺失**：CI 环境通常不包含 `local.properties` 文件（该文件通常由 IDE 生成且被 git 忽略），导致依赖此文件获取 Flutter SDK 路径的脚本失败。
  2.  **Git 环境不完整**：CI 环境可能使用浅克隆（shallow clone）或下载 ZIP 包方式获取代码，导致 `.git` 目录缺失或不完整，使得 `git ls-files` 等命令失败。
  3.  **Gradle 插件加载逻辑缺陷**：Flutter 工具链中的 `app_plugin_loader.groovy` 强依赖 `local.properties`，未考虑环境变量回退。

- **修复方案**：
  1.  **增强 app_plugin_loader.groovy**：修改 `_toolchain/flutter/packages/flutter_tools/gradle/src/main/groovy/app_plugin_loader.groovy`，添加逻辑：当 `local.properties` 不存在或未定义 `flutter.sdk` 时，尝试从 `FLUTTER_ROOT` 环境变量获取路径。
  2.  **优化预检脚本**：修改 `scripts/pre-check.sh` 和 `scripts/pre-check.ps1`，在执行 Git 跟踪检查（Check 8）前，先验证当前目录是否为有效的 Git 仓库。如果不是，则跳过该检查或仅输出警告，防止脚本因 `set -e` 或错误码非零而中断。
  3.  **统一环境变量回退策略**：确保 `android/settings.gradle` 和 `android/app/build.gradle` 中的 Flutter SDK 路径获取逻辑一致：优先读取 `local.properties`，若失败则回退到 `FLUTTER_ROOT` 环境变量。

#### 3. JDK 与 Gradle 版本兼容性
- **问题**：`android/gradle/wrapper/gradle-wrapper.properties` 指定了 **Gradle 8.14**，该版本要求 **Java 17** 或更高版本运行。
- **风险**：
  1.  `android/gradle.properties` 中硬编码了 `org.gradle.java.home=../_toolchain/jdk-17`。该路径下的 JDK 包含 Windows 二进制文件 (`.exe`, `.dll`)。
  2.  **在 Linux CI 环境下**，如果未覆盖此配置，Gradle 会尝试使用 Windows JDK 导致构建失败。
- **解决方案**：
  - **CI 配置建议**：在 CI 流水线执行 Gradle 命令时，必须通过命令行参数覆盖 `org.gradle.java.home`，例如：
    ```bash
    ./gradlew assembleRelease -Dorg.gradle.java.home=$JAVA_HOME
    ```
    或者确保 CI 环境的 `JAVA_HOME` 指向兼容的 Linux JDK 17+，并移除 `gradle.properties` 中的硬编码路径（推荐在 CI 脚本中处理）。

- 流水线同步策略：固定 CI Flutter 版本为 `3.27.1`，新增 Pub/Gradle 缓存加速，并在 `flutter pub get` 后通过 `grep .dart_tool/package_config.json` 自检 `amap_flutter_base/map` 是否指向 `plugins/` 目录，确保本地覆盖生效。

- 脚本兼容性策略：由于开发环境为 Windows，提交的 `gradlew` 可能带有 CRLF 行尾符或无执行权限。流水线配置 `chmod +x` 与 `sed -i 's/\r$//'` 确保脚本在 Linux Runner 上正确运行。
- 版本控制策略：修正 `.gitignore` 排除规则，确保 `gradle/wrapper/gradle-wrapper.jar` 与 `gradlew` 脚本被纳入版本控制，保证 CI 构建环境一致性。
- 兜底生效顺序：第三方插件可能在自身脚本中再次设置较低 `compileSdk`，因此兜底逻辑需要在项目脚本配置完成后再统一覆盖（通过 `gradle.afterProject`），以确保最终生效值为 35。
