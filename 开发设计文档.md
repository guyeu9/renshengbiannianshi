核心原则：**数据全部在本地**，没有自建后端，没有云数据库。联网只做两件事：调 AI 大模型 API、调 Embedding API。同步只走 WebDAV 网盘（如坚果云）[1]。

### 2.x Windows 本地开发环境（Flutter）

- Flutter SDK：放置在 `_toolchain/flutter`，并设置国内镜像环境变量：
  - `PUB_HOSTED_URL=https://pub.flutter-io.cn`

### 问题根因与修复（2026-02-23 CI 环境兼容性专项）

#### 1. CI 环境构建失败问题
- **现象**：CI 流水线（如 GitHub Actions）在预检或构建阶段失败，提示 `flutter.sdk` 未找到，或预检脚本报错。
- **根因**：
  1.  **local.properties 缺失**：CI 环境通常不包含 `local.properties` 文件（该文件通常由 IDE 生成且被 git 忽略），导致依赖此文件获取 Flutter SDK 路径的脚本失败。
  2.  **Git 环境不完整**：CI 环境可能使用浅克隆（shallow clone）或下载 ZIP 包方式获取代码，导致 `.git` 目录缺失或不完整，使得 `git ls-files` 等命令失败。
  3.  **Gradle 插件加载逻辑缺陷**：Flutter 工具链中的 `app_plugin_loader.groovy` 强依赖 `local.properties`，未考虑环境变量回退。

- **修复方案**：
  1.  **增强 app_plugin_loader.groovy**：修改 `_toolchain/flutter/packages/flutter_tools/gradle/src/main/groovy/app_plugin_loader.groovy`，添加逻辑：当 `local.properties` 不存在或未定义 `flutter.sdk` 时，尝试从 `FLUTTER_ROOT` 环境变量获取路径。
  2.  **优化预检脚本**：修改 `scripts/pre-check.sh` 和 `scripts/pre-check.ps1`，在执行 Git 跟踪检查（Check 8）前，先验证当前目录是否为有效的 Git 仓库。如果不是，则跳过该检查或仅输出警告，防止脚本因 `set -e` 或错误码非零而中断。
  3.  **统一环境变量回退策略**：确保 `android/settings.gradle` 和 `android/app/build.gradle` 中的 Flutter SDK 路径获取逻辑一致：优先读取 `local.properties`，若失败则回退到 `FLUTTER_ROOT` 环境变量。

#### 3. JDK 与 Gradle 版本兼容性
- **问题**：`android/gradle/wrapper/gradle-wrapper.properties` 指定了 **Gradle 8.14**，该版本要求 **Java 17** 或更高版本运行。
- **风险**：
  1.  `android/gradle.properties` 中硬编码了 `org.gradle.java.home=../_toolchain/jdk-17`。该路径下的 JDK 包含 Windows 二进制文件 (`.exe`, `.dll`)。
  2.  **在 Linux CI 环境下**，如果未覆盖此配置，Gradle 会尝试使用 Windows JDK 导致构建失败。
- **解决方案**：
  - **CI 配置建议**：在 CI 流水线执行 Gradle 命令时，必须通过命令行参数覆盖 `org.gradle.java.home`，例如：
    ```bash
    ./gradlew assembleRelease -Dorg.gradle.java.home=$JAVA_HOME
    ```
    或者确保 CI 环境的 `JAVA_HOME` 指向兼容的 Linux JDK 17+，并移除 `gradle.properties` 中的硬编码路径（推荐在 CI 脚本中处理）。
  - **已实施方案**：在 `.github/workflows/android-apk.yml` 和 `.github/workflows/android-pre-check.yml` 中添加了步骤，在构建前自动移除 `android/gradle.properties` 中的 `org.gradle.java.home` 配置，使 CI 使用 `setup-java` 配置的 JDK 17。

#### 4. AMap SDK 依赖冲突与修复
- **问题**：`android/app/build.gradle` 引入原生 `com.amap.api:search` 依赖时，出现两类构建失败：一是 `search:10.0.600` 在公共仓库无法解析；二是当使用 `search:9.x` 时与 `3dmap:10.0.600` 产生重复类冲突 (`Duplicate class com.amap.apis.utils.core.api.AMapUtilCoreApi`)。

- **修复方案**：移除 `android/app/build.gradle` 中未使用的 `com.amap.api:search` 依赖，保留 Web 搜索能力并避免依赖解析与重复类冲突。

### 2.y Flutter Gradle 配置迁移说明（2026-02-23）

#### 技术背景

Flutter 官方在 Flutter 3.16+ 推荐使用声明式 `plugins {}` 块，旧版命令式 `apply from`/`apply plugin` 方式已被弃用，未来版本将被移除。

#### 迁移内容

| 文件 | 旧版配置 | 新版配置 |
|------|---------|---------|
| **settings.gradle** | `apply from: "...app_plugin_loader.gradle"` | `pluginManagement {} + plugins { id "dev.flutter.flutter-plugin-loader" version "1.0.0" }` |
| **build.gradle** | `buildscript { dependencies { classpath ... } }` | 移除 `buildscript` 块，仅保留 `allprojects` 仓库 |
| **app/build.gradle** | `apply plugin: "com.android.application"` + `apply from: "...flutter.gradle"` | `plugins { id "com.android.application"; id "dev.flutter.flutter-gradle-plugin" }` |

#### 当前配置文件

- `android/settings.gradle`：使用 `pluginManagement` + `plugins {}` 块声明 Flutter 插件加载器和 AGP/Kotlin 版本
- `android/build.gradle`：仅保留 `allprojects` 仓库配置
- `android/app/build.gradle`：使用 `plugins {}` 块应用插件

#### 迁移收益

1. **消除弃用警告**：构建输出更清晰，无警告干扰
2. **未来兼容性**：避免未来 Flutter 版本移除旧版支持导致构建失败
3. **官方推荐**：符合 Flutter 3.16+ 最佳实践

#### 注意事项

- 新版配置中 `flutter-plugin-loader` 会自动应用 `flutter-gradle-plugin`，无需手动 `apply from: "...flutter.gradle"`
- `flutter {}` DSL 由 `flutter-gradle-plugin` 提供，配置块保持不变
- CI 环境需确保 `FLUTTER_ROOT` 环境变量正确设置
- 本地插件（如 `amap_flutter_location`）的 `build.gradle` 也需使用 `plugins {}` 块

- LinkDao 可测试性增强：UUID 生成从静态实例改为构造函数注入，默认仍使用 Uuid 实现，便于单测替换与控制生成结果。

## 四、年度报告系统设计（核心重点）

这是整个方案最关键的部分。PRD 要求年度报告包含"年度大事记、年度味觉地图、年度旅行足迹、年度友谊报告、年度情绪总结、年度目标完成情况" [1]。

### 4.1 核心原则：SQL 算数，AI 写文

AI 擅长写漂亮的感悟，但不擅长数数。所有统计指标必须由 SQL 预先算好，以"确定性数字"的形式喂给 AI，AI 只负责"基于这些数字写出有温度的文案"。

### 4.2 年度报告生成管线（5 步流水线）

```
Step 1          Step 2           Step 3          Step 4          Step 5
本地SQL      →  本地SQL       →  分模块调AI   →  汇总调AI    →  本地渲染
算统计指标      提取模块素材       生成模块文案      生成总报告      排版PDF
(纯本地)        (纯本地)          (联网API)        (联网API)      (纯本地)
```

### 4.3 Step 1：SQL 预聚合统计指标（纯本地，不耗 token）

```sql
GROUP BY poi_name HAVING revisit_count > 1
ORDER BY revisit_count DESC LIMIT 5;

GROUP BY month, mood;

SELECT mood, COUNT(*) as count
FROM moment_records
WHERE strftime('%Y', record_date)='2024' AND is_deleted=0
GROUP BY mood ORDER BY count DESC LIMIT 5;

-- 场景分布
SELECT scene_tag, COUNT(*) as count
FROM moment_records
WHERE strftime('%Y', record_date)='2024' AND is_deleted=0 AND scene_tag IS NOT NULL
GROUP BY scene_tag ORDER BY count DESC;

-- ========== 旅行模块统计 ==========
-- 足迹城市
SELECT DISTINCT city FROM travel_records
WHERE strftime('%Y', record_date)='2024' AND is_deleted=0 AND is_wishlist=0 AND city IS NOT NULL;

-- 旅行花费汇总
SELECT t.name as trip_name, t.start_date, t.end_date,
  SUM(tr.expense_transport) as transport,
  SUM(tr.expense_hotel) as hotel,
  SUM(tr.expense_food) as food,
  SUM(tr.expense_ticket) as ticket,
  SUM(tr.expense_transport+tr.expense_hotel+tr.expense_food+tr.expense_ticket) as total
FROM trips t
JOIN travel_records tr ON tr.trip_id = t.id
WHERE strftime('%Y', t.start_date)='2024'
GROUP BY t.id;

-- ========== 目标模块统计 ==========
-- 年度目标完成率
SELECT category,
  COUNT(*) as total,
  SUM(CASE WHEN is_completed=1 THEN 1 ELSE 0 END) as completed,ROUND(AVG(progress), 1) as avg_progress

JOIN moment_records m ON el.source_type='moment' AND el.source_id=m.id

-- 新认识的朋友
```

```dart
// 覆盖校验
Set<String> allIds = material.miniList.map((e) => e.id).toSet();
Set<String> coveredIds = {};

for (var batch in batches) {
  final response = await callAI(buildPrompt(stats, batch));
  coveredIds.addAll(response.coveredIds);
  runningSummary = response.runningSummary;
}

// 检查遗漏
Set<String> missed = allIds.difference(coveredIds);
if (missed.isNotEmpty) {
  // 补发遗漏的记录，让AI更新总结
  final missedItems = material.miniList.where((e) => missed.contains(e.id));
  final补充response = await callAI(buildSupplementPrompt(stats, runningSummary, missedItems));
  runningSummary = 补充response.runningSummary;
}
```

### 6.4 ChangeLog 自动记录机制

ChangeLog 在各 DAO 中自动记录，确保增量备份能追踪所有变更：

```dart
// DAO 集成示例
class FoodDao extends DatabaseAccessor<AppDatabase> with _$FoodDaoMixin {
  late final ChangeLogRecorder _changeLogRecorder = ChangeLogRecorder(db);
  
  Future<void> upsert(FoodRecordsCompanion entry) async {
    await into(db.foodRecords).insertOnConflictUpdate(entry);
    await _changeLogRecorder.recordInsert(
      entityType: 'food_records',
      entityId: entry.id.value,
    );
  }
  
  Future<void> softDeleteById(String id, {required DateTime now}) async {
    await (update(db.foodRecords)..where((t) => t.id.equals(id))).write(
      FoodRecordsCompanion(isDeleted: const Value(true), updatedAt: Value(now)),
    );
    await _changeLogRecorder.recordDelete(
      entityType: 'food_records',
      entityId: id,
    );
  }
}
```

### 6.5 数据恢复合并策略

恢复数据时采用"合并策略"，避免覆盖本地新数据：

1. **合并模式（默认）**：
   - 相同 ID 记录：取 updatedAt 较新者
   - 不同 ID 记录：全部保留
   - 删除标记：尊重删除状态

2. **覆盖模式**：
   - 清空本地数据，完全使用备份数据
   - 适用于设备丢失后恢复

### 7.50（原 6.49）本次实现说明（2026-02-23 Android 15 版本检测逻辑修复）

  - 增强版本检测逻辑，支持两种格式匹配：
    1. `Android 15` 格式（原有）
    2. 纯数字 `15` 格式（新增）
  - 如果两种格式都无法匹配，返回 `99`（保守假设为高版本）
  - 这样即使版本检测失败，也会安全地降级到 WebView 方案，避免崩溃
- **代码变更**：
  ```dart
  int? _androidMajorVersion() {
    if (!Platform.isAndroid) return null;
    final raw = Platform.operatingSystemVersion;
    var match = RegExp(r'Android\s+(\d+)').firstMatch(raw);
    if (match != null) {
      return int.tryParse(match.group(1) ?? '');
    }
    match = RegExp(r'^(\d+)').firstMatch(raw.trim());
    if (match != null) {
      return int.tryParse(match.group(1) ?? '');
    }
    return 99; // 保守假设为高版本，确保使用 WebView 方案
  }
  ```

  ```dart
    });
  ```

## 十、向量检索系统

### 10.1 系统概述

向量检索系统为应用提供语义搜索能力，通过将文本内容转换为向量表示，实现基于语义相似度的智能搜索，而不仅限于关键词匹配。系统采用云端 Embedding API 生成向量，将向量数据存储在本地 SQLite 数据库中，查询时在 Dart 侧进行余弦相似度计算。

核心特性：
- 支持多种实体类型的向量索引（美食、小确幸、旅行、目标）
- 后台异步处理向量索引任务，不阻塞用户操作
- 任务队列持久化和自动恢复机制
- 失败任务自动重试（最多 3 次，间隔递增）
- AI 服务不可用时优雅降级
- DAO 层自动触发向量索引

### 10.2 架构设计

```
                              │
                              ▼
```

### 10.4 服务层设计

#### 10.4.1 VectorIndexService

核心向量索引服务，提供以下功能：

- `indexRecord()`: 为记录创建向量索引
- `updateIndex()`: 更新记录的向量索引
- `deleteIndex()`: 删除记录的向量索引
- `findSimilar()`: 根据向量查找相似记录
- `searchByText()`: 根据文本搜索相似记录

#### 10.4.2 VectorIndexTaskQueue

后台任务队列服务，提供以下功能：

- `enqueue()`: 添加任务到队列
- `processQueue()`: 后台处理任务队列
- `loadPendingTasks()`: 应用启动时加载待处理任务
- `_executeTask()`: 执行单个任务
- `_handleTaskFailure()`: 处理任务失败，自动重试

**重试策略：**
- 最多重试 3 次
- 重试间隔递增：1s → 2s → 4s
- 超过最大重试次数后丢弃任务

#### 10.4.3 VectorIndexTrigger

向量索引触发器，封装与 DAO 层的集成：

- `recordInsert()`: 记录插入时触发索引
- `recordUpdate()`: 记录更新时触发索引更新
- `recordDelete()`: 记录删除时立即删除向量

**优雅降级：** 所有方法都包含 try-catch，异常时不影响主流程

#### 10.4.4 VectorIndexManager

统一管理器，整合所有向量索引相关服务：

- 工厂构造函数创建所有组件
- `initialize()`: 初始化任务队列
- 统一的 `recordInsert()/recordUpdate()/recordDelete()` 接口

### 10.6 后台任务队列

#### 10.6.1 任务流程

```
用户操作 (创建/更新记录)
    ↓
DAO 层触发
    ↓
VectorIndexTrigger.recordInsert/Update()
    ↓
VectorIndexTaskQueue.enqueue()
    ↓
任务加入队列 + 持久化到 SharedPreferences
    ↓
processQueue() 后台处理
    ↓
_executeTask() 调用 VectorIndexService
    ↓
调用 Embedding API 生成向量
    ↓
存储到 RecordEmbeddings 表
    ↓
任务完成，从队列移除
```

#### 10.6.2 任务恢复机制

应用启动时：
1. 调用 `VectorIndexManager.initialize()`
2. `VectorIndexTaskQueue.loadPendingTasks()` 从 SharedPreferences 加载待处理任务
3. 自动调用 `processQueue()` 继续处理未完成的任务
4. 所有 pending 和 failed 状态的任务都会被重新处理

### 10.7 配置与使用

#### 10.7.1 初始化流程

在 `AppShell` 的 `initState()` 中：
```dart
Future.microtask(() {
  ref.read(vectorIndexManagerInitializerProvider.future);
});
```

`vectorIndexManagerInitializerProvider` 会：
1. 获取 AppDatabase 实例
2. 调用 `db.initializeVectorIndexManager(embeddingServiceGetter)`
3. 初始化完成后，DAO 层即可使用向量索引功能

#### 10.7.2 Embedding 服务配置

向量索引依赖 `activeEmbeddingServiceProvider` 提供的 Embedding 服务，需要先在 AI 模型管理中配置可用的 Embedding 服务。

当前默认配置：
- 服务商：模力方舟（Gitee AI）
- 模型：Qwen3-Embedding-8B
- 备用模型：Qwen3-Embedding-4B → Qwen3-Embedding-0.6B → jina-embeddings-v4 → bge-m3

#### 10.7.3 优雅降级

如果 Embedding 服务未配置或不可用：
- DAO 层的记录操作正常完成
- 向量索引触发器捕获异常，不影响主流程
- 向量索引任务不会被创建
- 语义搜索功能不可用，但关键词搜索（FTS5）仍可使用

### 10.8 性能与优化

#### 10.8.1 余弦相似度计算

使用批量计算优化：
- 先批量加载所有向量
- 在 Dart 侧进行余弦相似度计算
- 按相似度排序后返回 Top N 结果

#### 10.8.2 混合搜索

`SemanticSearchService` 提供混合搜索能力：
- 结合语义搜索（向量相似度）和关键词搜索（FTS5）
- 可配置权重（默认 semantic: 0.7, keyword: 0.3）
- 综合排序返回最佳结果

#### 10.8.3 任务队列优化

- 异步处理，不阻塞用户操作
- 任务持久化，应用重启后自动恢复
- 失败重试机制，提高成功率
- 限制最大重试次数，避免无限重试

## 十二、架构决策记录（ADR）

### ADR-007: PDF导出服务彻底简化方案

- **背景**：
  1. PDF导出Null check operator错误依然存在，原方案仅添加异常处理未从根本解决问题
  2. _textStyle自定义方法虽然添加了异常处理，但在PDF库内部仍可能触发null check
  3. 复杂的UI组件（渐变、装饰器、统计卡片）在PDF库内部处理时可能触发额外的null检查
  4. 日志系统过于冗余，DEBUG级别日志大量输出到控制台造成信息干扰

- **冲突**：
  1. 原方案的异常处理策略未能彻底消除问题根源
  2. 需要在功能完整性和代码可靠性之间做出权衡
  3. 复杂的UI效果虽然美观，但增加了出错概率

- **决策**：
  1. **日志系统优化**：修改`_log`方法，只输出INFO、WARN、ERROR级别日志到控制台，DEBUG级别仅记录不输出
  2. **彻底简化PDF导出服务**：
     - 完全移除对自定义`_textStyle`方法的依赖，所有详情页直接使用`pw.TextStyle`
     - 简化`_createOverviewChapter`方法，使用纯文本列表替代复杂的统计卡片
     - 简化所有章节封面页，移除渐变效果，使用纯色背景
     - 逐个修复所有详情页的样式调用（美食、小确幸、羁绊、旅行、目标、时间线）
  3. **放弃原方案的原因**：原方案仅添加异常处理，但未能从根本上解决PDF库内部的null check问题

- **后果**：
  1. ✅ 消除了所有可能触发null check operator的代码路径
  2. ✅ 日志系统更加清晰，只输出关键信息
  3. ✅ 代码复杂度大幅降低，维护性提升
  4. ⚠️ 牺牲了一些视觉效果（如渐变、阴影、统计卡片），但保持了功能完整性

- **验证**：
  - [x] flutter analyze 无错误（仅有几个无关的未使用元素警告）
  - [x] 所有_textStyle调用已替换为pw.TextStyle
  - [x] 日志系统已优化，DEBUG级别不输出到控制台
  - [x] 所有复杂UI效果已简化

### ADR-006: 数据恢复快照回滚机制

- **背景**：恢复失败时无法回滚，可能导致数据丢失；用户在进行数据恢复操作时，如果恢复过程中出现错误，原有数据可能被部分覆盖，造成不可逆的数据损失
- **冲突**：需要额外的存储空间保存快照，但能显著提高数据安全性
- **决策**：
  1. 在恢复前创建本地快照，复制数据库文件到临时目录
  2. 记录媒体文件列表和快照元数据（createdAt, mediaFiles）
  3. 恢复失败时自动调用 `_restoreFromSnapshot()` 回滚到之前状态
  4. 快照使用时间戳命名，支持多个快照版本
- **后果**：
  - 恢复操作更安全，失败时可以自动回滚
  - 需要临时存储空间保存快照（通常几十MB）
  - 快照在恢复成功后可选择保留或清理
- **验证**：
  - 测试恢复失败场景，验证回滚功能正常工作
  - 验证快照包含完整的数据库文件和媒体文件列表
  - 验证回滚后数据与恢复前一致

### ADR-005: 外部存储导出与恢复方案

- **背景**：导出文件只能保存到应用内部目录，用户无法直接访问备份文件；Android 10+ 限制了外部存储访问权限
- **冲突**：一方面需要让用户能够方便地访问备份文件，另一方面需要遵守Android存储权限限制
- **决策**：
  1. 使用 `share_plus` 分享导出文件，用户可通过系统分享功能发送到其他应用或保存到任意位置
  2. 使用 `file_picker` 选择外部备份文件进行恢复，支持从任意位置选择 .zip/.json 文件
  3. 添加 `WRITE_EXTERNAL_STORAGE` 权限（Android 9及以下）
  4. 添加 `MANAGE_EXTERNAL_STORAGE` 权限（Android 11+）
  5. 配置 `requestLegacyExternalStorage`（Android 10兼容）
- **后果**：
  - 用户可通过系统分享功能导出文件到任意位置
  - 用户可从外部存储选择备份文件恢复
  - 兼容Android 9/10/11+不同版本的存储权限策略
- **验证**：
  - 测试JSON导出后可通过分享功能发送到其他应用
  - 测试本地备份完成后可分享备份文件
  - 测试从外部存储选择备份文件恢复功能正常工作

### ADR-004: UI 组件统一与设计规范建立

- **背景**：应用中存在 UI 组件样式不统一的问题：AI 解析按钮在不同页面样式不一致、部分弹窗仍使用原生 Android 弹窗、缺乏统一的 UI 设计规范文档
- **冲突**：一方面需要保持现有功能正常，另一方面需要建立可复用的组件库和规范文档
- **决策**：
  1. 创建可复用的 `AiParseButton` 组件，完全复用首页日程页面的 AI 史官按钮样式
  2. 创建可复用的 `CustomBottomSheet` 组件，包含拖拽指示器和顶部圆角 24px
  3. 替换所有删除确认弹窗为底部弹窗样式
  4. 建立完整的 `UI设计规范.md` 文档，包含色彩系统、按钮规范、卡片规范、弹窗规范等
  5. 确认并统一主题蓝色为 `#2BCDEE`
- **后果**：
  - 所有页面的 AI 解析按钮样式统一，提升视觉一致性
  - 弹窗样式统一，提升用户体验
  - 建立了完整的 UI 设计规范，便于后续开发复用
- **验证**：
  - 所有功能页面（美食、小确幸、旅行、目标、羁绊）都使用统一的 AiParseButton
  - 删除确认弹窗都使用 CustomBottomSheet 组件
  - UI 设计规范文档完整覆盖色彩、按钮、卡片、弹窗等规范

### ADR-001: 标签统计缓存方案变更

- **背景**：原计划新增 `_getTagCounts` 方法和 `_tagCountsCache` 缓存变量优化标签统计性能
- **冲突**：`flutter analyze` 报告 `_getTagCounts` 未被使用（`unused_element` 警告）；当前实现使用嵌套 `StreamBuilder` 实时监听数据库变化，缓存在这种场景下反而可能导致数据不一致
- **决策**：移除缓存代码，保持 `StreamBuilder` 的实时更新机制
- **后果**：标签统计通过嵌套 `StreamBuilder` 实时监听数据库变化，每次数据变化都会自动触发重新计算标签计数，无需额外缓存
- **验证**：
  - 进入"个人中心" → "模块管理"页面
  - 验证各模块标签列表正确显示使用次数
  - 验证未分类标签区域正确显示（橙色背景）
  - 点击未分类标签可添加到标签库
  - 删除标签时显示使用次数，可选择"替换"或"删除"
  - 替换标签后数据库中的标签数据正确更新
